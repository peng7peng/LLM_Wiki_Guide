# 社区优化建议

> 来自 Karpathy Gist 评论区的生产环境实战经验。在搭建完基础 Wiki 之后，Agent 逐条询问用户是否采纳。

## 来源

- **bluewater8008** 的团队在生产环境跑了几周后的总结（Gist 评论区）
- Karpathy 原文中的 Tips and tricks
- 实际搭建中踩坑经验的沉淀

## 优化项清单

### ① 分类提取

**问题**：50 页的论文和 2 页的随笔，用同样的方式处理，浪费 token 且效果差。

**方案**：在 schema 里定义资料类型，每种对应不同提取策略：

| 资料类型 | 处理方式 | 产出 |
|:--------|:---------|:-----|
| 概念/理论文章 | 提取核心定义和要点 | concept 页面 |
| 官方文档/API 文档 | 提取关键模式 | 更新已有页面 |
| 实战经验/踩坑记录 | 提炼教训和可复用模式 | experience 页面 |
| 项目复盘/总结报告 | 提炼目标-过程-结果-反思 | review 页面 |

**是否采纳？**

---

### ② 索引分级（L0-L4）

**问题**：Wiki 大了之后，每次 Query 都读完全部 index 会浪费大量 token。

**方案**：将 index.md 分为 4 级，渐进式披露：

| 级别 | 内容 | Token 预算 | 读取时机 |
|:-----:|:--------|:------------:|:-------|
| L0 | 一句话概览 | ~200 | 每次查询必读 |
| L1 | 按分类的目录 | 1-2K | 会话启动时读 |
| L2 | 带摘要的主题索引 | 2-5K | 按需读取 |
| L3 | 完整文章列表 | 5-20K | 深度搜索 |

**纪律：不读完索引就不读正文。**

**是否采纳？**

---

### ③ 三种实体模板

**问题**：概念页、经验页、复盘页的结构如果不确定，Agent 写出来的页面五花八门，后续维护困难。

**方案**：预定义三种页面的结构模板：

- **concept**：定义 → 要点 → 关联
- **experience**：背景 → 教训 → 可复用模板
- **review**：目标 → 过程 → 结果 → 反思

每种模板带 YAML frontmatter（domain / type / sources / created）。

**是否采纳？**

---

### ④ Query 双输出 ⭐ 最重要

**问题**：你问了一个问题，Agent 回答了你——这是输出。但回答过程中产生的新发现、新关联呢？如果不明确要求，这些有价值的东西就永远丢在聊天记录里了。

**方案**：在 schema 里写死——Query 时必须做两件事：
- 输出 A：回答问题
- 输出 B：把新发现写回 Wiki

这样你的每一次提问都在给知识库加料。

**是否采纳？**

---

### ⑤ 从第一天打标签

**问题**：Wiki 有几十个页面之后，你不知道哪些是枢纽节点。

**方案**：每个页面加 YAML frontmatter 标签：
```yaml
domain: llm, agent-loop
type: concept
sources: [xxx.md]
created: 2026-06-11
```

等 Wiki 大了之后，同时出现在多个 domain 下的共享页面就是最有价值的枢纽节点。

**是否采纳？**

---

### ⑥ 你是主编，不是甩手掌柜

**问题**：Agent 可以在没有来源的情况下瞎编综合结论，你不仔细看根本发现不了。

**方案**：两条纪律：
- 强制要求每句话都标来源
- 定期抽查 Wiki 内容

**LLM 是作者，你是主编。**

**是否采纳？**

---


## 使用方式

Agent 在完成基础搭建后，依次询问用户每条是否采纳。用户说"要"就写入 schema.md，说"不要"就跳过。也可以之后随时让 Agent 补充。
